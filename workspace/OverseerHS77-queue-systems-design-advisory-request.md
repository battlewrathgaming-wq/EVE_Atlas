# OverseerHS77 - Queue Systems Design Advisory Request

Date: 2026-05-25
Role: Atlas Overseer
Status: advisory request; not a Dev runway

## Decision

Do not open Dev for queue stale/expiration yet.

The Human-provided audit exposed a deeper architecture question: Atlas may need a clearer provider request/work queue model before stale/expiration, queue cadence, live-search priority, and Watch-driven freshness can be implemented safely.

## Why Systems Design First

The current queue table, `discovered_killmail_refs`, is a Discovery/provenance queue for killmail refs. It is not clearly a generalized provider request/work queue.

Human intent now includes:

- Watch-scheduled work added on scoped intervals
- live searches entering queue with priority instead of pulsing provider calls directly
- freshness tied to Watch interval/lookback
- uniqueness per Watch configuration/target/time/scope where appropriate
- local hydration excluded from queue policy

Those requirements may require a design split between:

- Watch schedule/work requests
- live search priority requests
- provider discovery refs
- ESI expansion refs
- local hydration jobs
