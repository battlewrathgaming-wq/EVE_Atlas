## Decision

HS72 is accepted after one Overseer hardening correction.

Atlas now has bounded runtime snapshot destination and snapshot/support-artifact budget authority. This is accepted as support/recovery infrastructure only. It is not Evidence, not Discovery, not Assessment Memory, not production deletion, not active DB relocation, not restore, and not automatic cleanup.

## Accepted Behavior

- Runtime snapshot settings are backend-owned and versioned.
- Atlas can persist a validated snapshot destination directory.
- Atlas can persist a snapshot/support-artifact budget.
- Missing settings fall back to the project `.tmp/db-snapshots` path.
- Invalid persisted destination or budget settings degrade visibly and are not silently used.
- Snapshot filenames are generated by backend snapshot logic.
- Snapshot preflight reports current destination usage, projected snapshot bytes, projected usage, configured budget, remaining budget, and over-budget blockers.
- Snapshot creation is blocked when projected usage exceeds the configured budget.
- `automatic_cleanup` remains false; Atlas does not prune or delete old snapshots/support artifacts to satisfy the budget.
- Existing explicit snapshot creation confirmation remains preserved.

## Overseer Correction

Review found a renderer authority gap: renderer-origin service calls could provide a payload-level settings path for snapshot settings get/update and preflight.

Correction applied:

- Renderer-origin snapshot settings get/update now use the context-owned backend settings path when supplied.
- Renderer-origin snapshot preflight ignores payload-supplied settings paths.
- Non-renderer/internal callers may still pass explicit settings paths for tests and controlled backend flows.
- Verification now covers renderer-supplied settings path rejection/ignore behavior.

This keeps the accepted boundary: renderer code can request snapshot status/settings through controlled service commands, but cannot choose arbitrary filesystem paths.
