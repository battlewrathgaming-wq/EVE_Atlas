# Observation / Relationship Pivot

Status: Advisory discovery input, not project authority
Surface: Observation / relationship pivot
Source owner: AURA-Atlas
User task: Start from a known anchor and see a connected local story across stored Evidence/EVEidence, labels, Assessment context, and provenance.
Source data / command: Observation lookup model, relationship pivot advisory, actor/radius/corporation reports, `activity_events`, `killmails`, local lookup tables.
What is visible now: The data substrate can support pivots by killmail, pilot, corporation, system, ship/type, role, and time, but the product surface is not final.
What the user needs to understand: Observation assembles connected records into a story over stored local Evidence/EVEidence; it is not new truth, hidden assessment, or proof of current state.
First-read candidates: Seen in stored Evidence; related records; event-time corporation context; local story basis; labels incomplete; Discovery-only leads nearby.
Detail/diagnostic candidates: Anchor ID, evidence count, time window, related pilots/corporations/systems/ships, role split, raw IDs, label completeness, Discovery-only refs, Assessment citations, provenance.
Terms to preserve: Observation, Evidence/EVEidence, Discovery, Assessment Memory, anchor, relationship pivot, event-time context.
Terms to avoid or qualify: proof, current membership, live location, complete activity, intent, threat certainty, automatic intelligence.
Boundary that must not blur: Observation is story/query over connected records and does not create Evidence/EVEidence or Assessment.
Risks / false implications: Relationship graphs can overclaim causality; local records can look complete; corp context can be mistaken for current roster; Assessment context can look like proof.
Possible request_display candidate: yes
No Dev authorization: This note does not authorize schema changes, relationship graph implementation, provider lookups, Observation read model persistence, or final UI design.
