Accepted direction from discussion:

- Support artifacts are helper files for inspection, recovery, debugging, or proof of behavior.
- Support artifacts are not Evidence/EVEidence, Discovery refs, Observation, Assessment Memory, or product reports unless Atlas explicitly defines a separate product surface.
- Light operational app logs before storage setup are acceptable if bounded.
- Real corpus snapshots are blocked before storage setup because a real corpus should not exist yet.
- Fixture/demo snapshots remain separate.
- Snapshots and cache/recovery material should be protected from ordinary one-step cleanup and belong in a second-stage/recovery cleanup path.
- Light app-running noise is not the main storage concern.
- Corpus-adjacent material counts against the storage/corpus budget: scanning, Discovery, Evidence/EVEidence, Hydration labels, Assessment-linked output, trace packs, snapshots, support exports, and anything generated by real Atlas activity.
- External I/O controls provider contact and provider-derived movement.
- Storage authority controls local persistence.
- Assessment and Watch configuration are local writes, not External I/O actions.
- Support artifacts follow local/storage rules unless they would trigger provider contact, which they must not.

## Goal

Add a read-only support-artifact path authority inventory.

The inventory should make Atlas able to describe support artifact classes, path posture, budget posture, External I/O posture, renderer safety, cleanup stage, and sensitivity without creating, moving, deleting, restoring, uploading, or packaging files.

## Required Output Shape

Add a service command/readout if appropriate, likely:

```text
support.artifact_path_authority.preview
```

The readout should include representative support artifact classes such as:

- runtime snapshots
- trace packs
- logs
- readiness/preflight reports
- temp/cache files
- SDE/import source or derived lookup material
- config/write-proof fixture artifacts if already represented locally

For each class, report:

- artifact class/name
- family: `operational_support` or `corpus_adjacent_support`
- current or candidate path basis, when computable
- whether it is allowed before storage setup
- whether it requires storage authority
- whether it counts against storage/corpus budget
- whether it is local-only or provider-capable
- External I/O relevance
- renderer safety / trusted-context requirement
- cleanup stage: ordinary cleanup, recovery cleanup, fixture-only, or blocked
- privacy/sensitivity posture
- rolling vs retained snapshot posture where relevant
- whether the packet is read-only and non-mutating

## Guardrails

- No support artifact creation.
- No snapshot creation.
- No trace-pack creation.
- No cleanup/delete/prune/restore/move/copy/migration.
- No provider calls.
- No External I/O enforcement.
- No runtime command blocking.
- No storage config write.
- No schema changes.
- No UI/renderer redesign.
- No filesystem probing from renderer-provided arbitrary paths.
- Do not treat support artifacts as Evidence/EVEidence.
- Do not treat trace packs as evidence export.
- Do not make cache a single policy bucket; classify cache by origin when possible.
- Do not make snapshots invisible hidden storage.
- Do not open cleanup execution.

## Stop Conditions

Stop and return to Overseer if:

- the proof needs to create, delete, move, restore, or package files
- the proof needs to call providers or inspect live/private remote state
- existing path state cannot be inspected without mutating the filesystem
- renderer payloads would need to choose arbitrary paths
- runtime schema changes are needed
- support artifact classes cannot be separated from Evidence/EVEidence or Assessment meaning
- cleanup policy requires destructive execution decisions
