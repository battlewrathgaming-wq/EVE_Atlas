## Review Result

HS480 is accepted.

The artifact answers the requested Architecture/Data design pass and preserves the current Atlas boundary model:

- Watch emits bounded work and owns cadence interpretation.
- Discovery owns provider-facing movement and provider policy facts.
- External I/O gates Discovery/provider movement, not Watch emission.
- `fetch_runs` is not the Watch bucket.
- `discovered_killmail_refs` is not the pre-acquisition Watch bucket.
- Evidence/EVEidence remains landed ESI killmail truth, not Watch scheduler/completion state.
- Provider timing facts may cross into receipts, but Watch scheduling decisions do not.

No code, schema, provider call, Discovery pickup, candidate ref write, Evidence/EVEidence write, Hydration, Observation, dispatcher/lease runtime, UI, source-term rename, or protected-word update was performed by the advisory.

## Accepted Direction

Atlas should add a small product Watch bucket persistence surface for emitted Watch work identity before opening Discovery pickup execution.

The first implementation should prove durable identity and restart-safe duplicate suppression only.

Accepted next shape:

```txt
Watch due/stub basis
-> one open product bucket item per Watch
-> durable accepted scope/window/cap snapshot
-> no provider movement
-> no Discovery refs
-> no Evidence/EVEidence writes
-> no Watch cadence mutation
```

## Overseer Decisions For Next Packet

1. First product bucket table:
   - Use a generic Watch bucket table shape, but the first writer/verification scope is `system_radius` only.
   - Actor Watch migration remains parked.

2. First product command/source:
   - Create rows from validated Watch-run stub/emission basis, not arbitrary renderer input.
   - The implementation may source that basis from stored Watch rows only through an explicit trusted service path.

3. Alpha status names:
   - Accept `open`, `settled`, `cancelled`, `blocked_integrity`.
   - Do not use `held_by_external_io` as a bucket lifecycle status.

4. `watch_run_id`:
   - Generated by the trusted Watch emission/bucket service from accepted Watch/stub basis.
   - Treat format as opaque local implementation detail, not public API.

5. Receipt/provider fields:
   - Nullable schema fields are acceptable if they remain inert in the first packet.
   - No settled receipt mutation is opened in the first packet.

6. Provider timing facts:
   - JSON-only for alpha is acceptable.
   - These facts remain provider/Discovery facts, not Watch cadence authority.

7. Minimum readout:
   - Prove one open row per Watch.
   - Prove stale missed intervals collapse to one open row.
   - Prove overlapping Watch scopes coexist under different Watch IDs.
   - Prove External I/O off does not block bucket row creation.
   - Prove External I/O off creates zero provider packets, Discovery refs, Evidence/EVEidence writes, and Watch cadence mutations.

## Caveats

The next Dev packet may include schema/repository/service work, but must not combine bucket persistence with Discovery pickup execution, leases, dispatcher behavior, provider movement, candidate refs, Evidence/EVEidence writes, Watch cadence mutation, UI, or collector retirement.

The design should be treated as product schema direction, not final naming doctrine for every future Watch/Discovery/receipt field.
