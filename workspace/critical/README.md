# Critical Terms And Assets

This folder records Atlas-critical terms and assets while they are active coordination objects.

Use these files when work touches Atlas emitted meanings, bridge-facing payloads or labels, source/evidence/assessment terminology, critical workflows, or downstream presentation interpretation.

Accepted terminology policy:

- `F:\Projects\Docs\Aura-Project-Orchestration\terminology\TerminologyAuthorityRuleset-2026-05-24.md`

Authority:

- Atlas owns what Atlas emits and what it means.
- Atlas owns its internal language and its Project -> Bridge language.
- Lab owns human-facing presentation after the bridge, but must preserve Atlas meaning unless Human override resolves a conflict.
- Lab should use its own product-agnostic presentation vocabulary for Lab defaults; that vocabulary must not rewrite Atlas emitted meaning.
- Shared spelling does not imply shared meaning.

Bridge boundary:

- Project internal language is owned by the project.
- Project -> Bridge language is owned by the emitting project.
- Bridge-facing means project-owned language, labels, states, payloads, or events exposed outside the project for downstream bridge or presentation use.
- Bridge -> Interface language is Lab-owned presentation language, bounded by source fidelity and Human decisions.

Files:

- `critical-assets.md` lists Atlas files, workflows, payloads, modules, or docs agents should treat carefully.
- `critical-terms.md` lists Atlas-owned meanings, preserve-exact terms, and downstream translation cautions.
- `misc/` is a fallback for important notes that do not yet fit the two main files.

Overseer only: when the Human gives authority, the project Overseer may update this local critical folder or another accepted local resting-state record. Specialists, Dev, and advisory workers should flag the issue in their handoff instead of updating authority surfaces themselves.
