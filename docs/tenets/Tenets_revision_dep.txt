## AURA Atlas Tenets

AURA Atlas is an evidence-driven intelligence system, not a UI-driven dashboard.

The UI exists to:

* present
* slice
* filter
* inspect
* correlate
* explain

stored evidence.

The UI must not become the place where intelligence facts are invented, mutated, or made authoritative.

AURA-7 observes now.
AURA Atlas remembers, scopes, and explains.

The core promise:

Where killmail evidence becomes operational memory.

1. Evidence First

---

The durable source of truth is the expanded ESI killmail.

Pipeline:

zKill discovery
→ ESI expanded killmail
→ stored evidence
→ normalized events
→ derived reports

zKillboard is discovery only.

zKill may provide:

* killmail_id
* zkb.hash

But tactical truth comes from expanded ESI killmails.

Store expanded ESI killmails once.
Recompute everything else from them.

2. Immutable Evidence Layer

---

Expanded ESI killmails should be treated as immutable evidence records.

Corrections, enrichments, annotations, and derived intelligence should layer on top of evidence rather than mutating the original evidence object.

Evidence records should remain:

* append-oriented
* reproducible
* auditable
* rebuildable

3. IDs Are Facts, Names Are Labels

---

Evidence records preserve original numeric IDs:

* killmail_id
* character_id
* corporation_id
* alliance_id
* ship_type_id
* weapon_type_id
* solar_system_id

Names are cached display metadata.

Names improve readability, but they do not replace IDs.

Reports may show:

Armageddon [typeID: 643]
ZTS-4D [solarSystemID: 30004660]
The Initiative. [allianceID: 1900696668]

The ID remains the fact.
The name is presentation.

4. UI Is Presentation, Not Authority

---

The interface should behave like a set of slicers over stored evidence.

Examples:

* system slicer
* radius slicer
* actor slicer
* corporation/alliance slicer
* time-window slicer
* role slicer: attacker/victim
* geography slicer: system/constellation/region
* disposition slicer: friendly/neutral/hostile/ignored
* ship/group/category slicer

The UI may:

* filter
* summarize
* correlate
* visualize
* explain

But it should not mutate evidence meaning.

Good UI behavior:

* Show stored evidence matching this scope.
* Explain the time window and sample size.
* Allow drill-down to raw evidence.
* Show provenance and uncertainty.

Bad UI behavior:

* Treat zKill summaries as authoritative.
* Create counters that cannot be rebuilt.
* Hide or erase evidence from storage.
* Allow AI summaries to become evidence.
* Replace IDs with mutable names internally.

5. Observation Is Not Interpretation

---

Observed behavior is not intent.

Repeated appearances:

* in a system
* with a corporation
* alongside another pilot
* within a region

are observations.

Interpretations such as:

* “staging system”
* “home region”
* “coalition”
* “response fleet”
* “hunting pattern”

must remain clearly labeled as analysis or commentary.

AURA Atlas stores:

* evidence
* observations
* derivations

Interpretation must remain distinguishable from fact.

6. Collection Provenance Is Not Intelligence Scope

---

A report should usually be scoped by what the evidence describes:

* system + time
* radius + time
* actor + time
* region + time

not by how the evidence was collected.

Collection metadata is provenance:

* run_id
* discovery route
* collection timestamp
* API counts
* warnings
* capped/partial status

Use run reports to answer:

“What happened during this collection run?”

Use intelligence reports to answer:

“What evidence exists for this scope?”

7. Scope And Sample Size Must Be Explicit

---

Every report must explain what it is based on.

At minimum:

* evidence window
* systems/actors included
* stored killmail count
* activity event count
* discovered refs if relevant
* expanded sample count
* partial/complete status
* source statement
* build timestamp

Example:

Stored evidence matching this scope:
8 killmails / 50 activity events.

Evidence window:
2026-05-21T00:00:00Z
→ 2026-05-22T00:00:00Z

Collection provenance may include multiple run types.

Source:
zKill discovery + ESI expanded killmails.

Reports must not sound more certain than the evidence allows.

8. Collection Is Incomplete By Default

---

Collection coverage is probabilistic, not complete.

Absence of evidence is not evidence of absence.

The system should assume:

* polling gaps exist
* zKill visibility is partial
* ESI expansion may fail
* rate limits may truncate collection
* time windows may miss events

Unless explicitly proven otherwise, reports should assume partial visibility.

9. Derived Intelligence Must Be Rebuildable

---

Actor timelines, heatmaps, activity rankings, corp frequency, footprints, timelines, and relationship views must be derived from stored evidence.

Derived data may be:

* cached
* indexed
* accelerated
* summarized

But it must remain rebuildable.

If a number cannot be traced back to evidence, it should not become authoritative intelligence.

10. Derived Views Are Disposable

---

Derived analytics are accelerators, not evidence.

Examples:

* heatmaps
* operator rankings
* activity scores
* threat indexes
* hotspot summaries
* association graphs

may be cached for performance, but they should always be discardable and rebuildable from stored evidence.

11. Entity Identity Is Temporal

---

Entity relationships are time-scoped.

The following may change over time:

* corporation membership
* alliance affiliation
* operational association
* naming
* activity patterns

Historical relationships should be preserved historically rather than overwritten globally.

12. Disposition Filters Do Not Delete Evidence

---

Whitelist/friendly/ignored status affects:

* presentation
* ranking
* filtering
* alerting

It must not:

* prevent ingestion
* erase evidence
* mutate stored records

A friendly entity appearing in a killmail is still evidence.

The UI may:

* label
* suppress
* de-prioritize

but storage remains complete.

13. AI Is Commentary, Not Evidence

---

AI may:

* summarize
* narrate
* explain
* correlate
* assist interpretation

AI may not become:

* the evidence source
* authoritative fact
* hidden transformation logic

AI output should:

* remain labeled as commentary
* cite evidence scope
* link back to underlying evidence where possible

14. Defensive API Orchestration

---

AURA Atlas is a slow-burn intelligence system, not a high-speed scraper.

The goal is:

* persistent evidence collection
* gradual enrichment
* scoped discovery
* sustainable observation

not:

* aggressive ingestion
* broad scraping
* low-value flooding
* real-time dominance

Collection should remain:

* conservative
* cache-aware
* deduplicated
* observable
* rate-aware

Repeated small observations over time are more valuable than aggressive broad collection bursts.

The platform should prioritize:

* evidence quality
* rebuildability
* sustainable polling
* low operational footprint

Collection should originate from:

* watched systems
* watched entities
* scoped tactical searches

not from:

* broad universe-wide scraping
* uncontrolled traversal
* mass polling

15. Watch-Based Discovery

---

System watches intentionally allow actors to expose themselves through repeated appearances.

This is:

* evidence accumulation
* not exhaustive surveillance

Watchlists answer:
“What is this actor doing over time?”

System watches answer:
“Who is active here?”

Repeated appearances become:

* observations
* patterns
* operational signals

not automatic conclusions.

The system should support:

* gradual discovery
* recurring exposure
* promotion from observed actor → watchlist entity

16. Polling Must Remain Conservative

---

Recommended defaults:

* hourly polling
* staggered schedules
* bounded expansion counts
* capped collection windows
* dedupe before expansion
* concurrency limits
* retry/backoff

The system should prefer:

* continuity
* sustainability
* observability

over:

* maximal ingestion speed

The system should degrade gracefully under:

* rate limiting
* API failures
* partial collection
* cache-only operation

17. Local-First Classification And Resolution

---

Use local/static metadata whenever possible.

Maintain local lookup metadata for:

* systems
* constellations
* regions
* stargate topology
* ship types
* groups
* categories
* static classifications

Examples:

* systems.json
* ships.json
* SDE-derived group/category metadata

Classification and naming should prefer:

* local SDE-derived lookup
* cached resolved entities

before:

* live ESI enrichment

Live API calls should primarily exist for:

* dynamic activity evidence
* entity discovery
* affiliation enrichment when unknown
* expanded killmail retrieval

The platform should avoid repeated live lookups for:

* static geography
* ship classifications
* ship names
* group/category metadata
* already-known entity labels

Classification still matters.

Ship/group/category classification is necessary for:

* doctrine analysis
* ship-role filtering
* heatmaps
* threat weighting
* logistics/capital detection
* composable slicers

But classification should come primarily from:

* local SDE-derived metadata
  not repeated live API calls.

18. Cache-Aware Enrichment

---

Existing knowledge should be reused before requesting additional enrichment.

Prefer:
cache hit
before:
live lookup

Examples:

* known corp IDs
* known alliance IDs
* immutable expanded killmails
* cached name resolutions
* local ship/group/category metadata

If useful metadata is already available during an expansion flow, it is acceptable to preserve and normalize it rather than discarding it and later re-requesting it.

However:
“available during expansion”
does not justify:
unbounded enrichment chains.

Expansion should remain scoped and bounded.

19. Respectful API Use

---

External API behavior must remain:

* conservative
* scoped
* observable
* rate-aware

Rules:

* zKill only for discovery refs
* ESI expansion only for uncached killmails
* dedupe before expansion
* cap refs and expansions
* use clear User-Agent
* retry/backoff respectfully
* log API counts and warnings
* prefer local SDE metadata over live calls where possible

Repeated small runs are preferable to aggressive broad collection.

20. Topology Is Lookup Metadata

---

SDE geography and type data are lookup/reference metadata, not intelligence evidence.

Use SDE for:

* systems
* constellations
* regions
* stargate adjacency
* ship/type names
* group/category metadata

This supports:

* slicers
* routing
* presentation
* classification

But activity facts still come from expanded killmails.

21. Slicers Should Compose Predictably

---

Combining slicers should narrow evidence scope transparently.

Composable filters may include:

* region
* time window
* ship class
* watchlist membership
* disposition
* attacker/victim role
* corp/alliance affiliation
* ship group/category

Slicers should not silently change analytical meaning.

The user should always understand:

* what evidence remains in scope
* why evidence was included
* why evidence was excluded

22. Provenance Must Travel With Reports

---

Derived reports must carry sufficient provenance metadata to reconstruct:

* evidence scope
* collection conditions
* source paths
* warnings
* caps/truncation
* build conditions

This is especially important for:

* exports
* screenshots
* AI summaries
* shared reports
* historical comparisons

23. Product Identity

---

AURA-7 is the tactical viewport.

AURA Atlas is the persistent evidence map.

AURA-7:

* observes
* alerts
* scans
* reacts

AURA Atlas:

* remembers
* scopes
* correlates
* explains

The HUD answers:
“What is happening now?”

Atlas answers:
“What patterns emerge over time?”
